# Exploration: first-release (0.1.0)

> Bump from `0.0.1` (placeholder published the day the npm name was reserved) to `0.1.0` (first real MVP release). MVP user-facing surface is complete: `init`, `run`, `list`, `clean`, `config`, with SIGINT cleanup proven end-to-end and a fuzzy branch picker.

## Current State

- **package.json**: `name: "@tanisjam/peel"`, `version: "0.0.1"`, `description: "Ephemeral git worktrees with auto-managed dev servers. Work in progress."`, `bin.peel: "./dist/index.js"`, `files: ["dist"]`, `engines.node: ">=20"`.
- **scripts**: `release: "changeset publish"`, `prepublishOnly: lint + typecheck + test + build` (already in place — gates run before any publish).
- **Changesets**: configured under `.changeset/config.json` with `access: public`, `commit: false`, `baseBranch: main`. No pending changesets exist.
- **CI** (`.github/workflows/ci.yml`): runs lint+typecheck+build+test on PR/main across Ubuntu+macOS × Node 20+22. **Does NOT publish** — there's no release workflow yet.
- **version source**: hardcoded in `src/version.ts` (`export const VERSION = "0.0.1"`). The CLI's `--version` flag is wired to this constant via `commander.version(VERSION)` in `src/index.ts`. The unit test (`src/version.test.ts`) only checks the constant is a non-empty string — bumps don't break tests.
- **CHANGELOG.md**: does not exist. README mentions Changesets but no changelog has been generated yet (the placeholder publish skipped that flow).

## Affected Areas

- `package.json` — `version` and `description`.
- `src/version.ts` — bump constant to match.
- `.changeset/<random>.md` — new changeset describing 0.1.0.
- `CHANGELOG.md` (NEW) — generated by `changeset version` from the changeset(s).
- `README.md` — drop "Work in progress / placeholder" wording from the top now that this is a real release; keep the install snippet pointing at `@tanisjam/peel`.

## Release path options

### A) Local manual publish (recommended for the FIRST release)

1. Add a changeset (`.changeset/<name>.md`) categorizing as `minor` (because 0.0.x → 0.1.0) and listing the user-visible features.
2. Run `npx changeset version` — bumps `package.json` to 0.1.0, generates `CHANGELOG.md`, deletes the consumed changeset file.
3. Manually update `src/version.ts` to match the new `package.json` version (the only source-of-truth divergence; see follow-up below).
4. Run `npm run prepublishOnly` (already wired) for last-mile verification.
5. Run `npm publish --access public` — uses the user's npm token from `~/.npmrc`.
6. Tag the commit (`git tag v0.1.0`) and push tags.
7. Open a GitHub release tied to the tag, pasting the CHANGELOG section (manually, or via `gh release create v0.1.0 --notes-file CHANGELOG.md` after trimming).

Pros: no GitHub-side secrets needed; full transparency; works today.
Cons: each release requires the maintainer to be at a machine with npm auth. Acceptable for a single-maintainer hobby CLI.

### B) Automated via `changesets/action` GitHub workflow

1. Add `.github/workflows/release.yml` running `changesets/action@v1` on push-to-main.
2. The action either opens a "Version Packages" PR (when changesets exist) or runs `changeset publish` (when versioning PR is merged).
3. Requires `NPM_TOKEN` GitHub secret with publish rights.

Pros: hands-off after merge; standard pattern.
Cons: introduces infra to debug on a release that's already a minor risk; needs a GitHub secret. Better suited for the SECOND release once we've shipped one manually and can confirm the CHANGELOG flow.

## Approaches

1. **Path A only for 0.1.0; Path B as a follow-up change** *(recommended)* — manual publish for the first release; document the steps in apply-progress; add Path B as a SUGGESTION for the next release.
   - Pros: minimal new infra; smallest blast radius; gives us a clean baseline tag (v0.1.0) before we automate.
   - Cons: re-doing the same dance for 0.2.0, 0.3.0, etc.
   - Effort: Low.

2. **Path B in this change** — release workflow + 0.1.0 publish in one PR.
   - Pros: hands-off forever after.
   - Cons: doubles the surface area of THIS change; a release-workflow bug + a first-release wobble in the same PR is harder to bisect.
   - Effort: Medium.

3. **Path A for 0.1.0 + add Path B in the same PR but DISABLED until 0.2.0** — write the workflow, gate it on `if: false` or comment it out, document the unlock step.
   - Pros: ships infra alongside the bump but doesn't activate it.
   - Cons: dead code committed; user must remember to flip the switch later.
   - Effort: Low-Medium.

## Recommendation

**Approach 1.** Manual `npm publish` for 0.1.0; capture every step in `apply-progress.md` so the next release is a 5-minute checklist; add the GitHub Action automation as a separate follow-up change after we know the manual flow produces what we want.

## Capability impact

**Zero.** This is a metadata + packaging change. No spec adds/modifies/removes. README copy and CHANGELOG updates are docs, not behavior. The proposal will declare "None" under both spec buckets.

## Risks

- **`src/version.ts` drift from `package.json`** — manual sync only. Mitigation: a one-liner unit test that asserts they match would catch this; not in scope for 0.1.0 but worth a SUGGESTION. Quickest immediate guard: extend the existing `src/version.test.ts` with `expect(VERSION).toBe("0.1.0")` (locks the bump) — small but rigid; I lean toward NOT adding it because every release has to update the test too. Better to defer to a "read package.json at build time" follow-up.
- **Stale README** — the top now says "Work in progress. The name `peel` on npm is squatted, so this is published as `@tanisjam/peel` — the binary on your terminal is still `peel`." Trim "Work in progress" but keep the scoped-name explanation.
- **Publish requires npm auth** — fails fast if the user isn't logged in (`npm whoami`); apply-progress should call this out as a precondition.
- **Changeset content drift** — the changeset is the SOURCE for CHANGELOG.md once consumed. We have to summarize 5 prior cycles into one minor-bump entry; the apply-progress should include the final approved text.
- **Unpublishable due to lockfile / build artifact** — `prepublishOnly` already runs the full pipeline; if anything's broken we know before `npm publish` is hit.
- **GitHub release tag** — manual `git tag v0.1.0 && git push --tags` after publish so the changelog has a permalink.

## Ready for Proposal

Yes. Single-cycle change with no spec impact. Two main artifacts produced: (1) a changeset file describing 0.1.0, (2) a generated `CHANGELOG.md` after `changeset version`. Two source-of-truth files updated (`package.json`, `src/version.ts`). One README polish. Then publish + tag.
