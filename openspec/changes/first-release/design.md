# Design: first-release (0.1.0)

## Technical Approach

Sequenced metadata change driven by Changesets. Author one minor changeset → run `npx changeset version` (auto-bumps `package.json`, generates `CHANGELOG.md`, deletes the consumed `.md`) → manually sync `src/version.ts` → trim README intro → `npm run prepublishOnly` for last-mile gates → commit + PR. After merge + maintainer's `npm publish`: tag `v0.1.0`, push tags, create GitHub release.

## Architecture Decisions

### Decision: Single minor bump (0.0.1 → 0.1.0), not a patch

**Choice**: Mark the changeset as `minor`.
**Alternatives considered**: `patch` (0.0.1 → 0.0.2).
**Rationale**: 0.0.x signaled "placeholder". 0.1.0 signals "real but pre-1.0". Future feature additions (e.g. `peel update`) can be `0.x.0` minor bumps; bug fixes go to `0.x.y` patch. Aligns with semver-zero conventions.

### Decision: Manual sync of `src/version.ts` (not auto-generated)

**Choice**: Keep `src/version.ts` as a hand-edited constant; bump it as a manual step in apply-progress.
**Alternatives**: (a) tsup `define`/banner injecting `package.json` version; (b) read `package.json` at runtime via `import` assertion.
**Rationale**: This is the FIRST release. The cleanest auto-sync (option a) is a follow-up — adding it in the same PR confuses the bisect surface. Option b breaks ESM `--noEmit` strictness. Manual is fine for one bump; document the drift risk; address in a dedicated change.

### Decision: Tag + GitHub release happen AFTER `npm publish` succeeds

**Choice**: Sequence the post-merge ops as (1) maintainer runs `npm publish`, (2) maintainer reports success, (3) orchestrator runs `git tag v0.1.0 && git push --tags`, (4) `gh release create v0.1.0 --notes-file <CHANGELOG section>`.
**Alternatives**: tag + release on merge, before publish.
**Rationale**: A tag pointing at "the version on npm" should match reality. If `npm publish` fails (auth, registry hiccup), we don't want a published GitHub release that links to a non-existent npm artifact. Sequencing publish-first protects from that.

### Decision: Generate CHANGELOG via the changeset, don't hand-write

**Choice**: Let `changeset version` produce `CHANGELOG.md`. The hand-written content lives only in the changeset body.
**Alternatives**: write `CHANGELOG.md` directly.
**Rationale**: Consistency with the tool's expected workflow; future bumps use the same pipeline; one source of truth (the changeset body) per release.

## Data Flow

```
.changeset/<random>.md ── changeset version ──┬──→ package.json (version bump)
        (input)                               ├──→ CHANGELOG.md (append entry)
                                              └──→ delete consumed .md
                                              
src/version.ts (manual edit) ─────────────────────────────→ matches package.json

PR merged ──→ user `npm publish` ──→ user reports success ──→ orchestrator:
                                                                git tag v0.1.0
                                                                git push --tags
                                                                gh release create v0.1.0
```

## File Changes

| File | Action | Description |
|------|--------|-------------|
| `.changeset/release-0-1-0.md` | Create | Source for version bump + CHANGELOG entry |
| `package.json` | Modified | `version: "0.0.1" → "0.1.0"`; `description` drops "Work in progress" |
| `src/version.ts` | Modified | `VERSION = "0.1.0"` |
| `CHANGELOG.md` | Create | Auto-generated by `changeset version` |
| `README.md` | Modified | Replace "Work in progress / placeholder" paragraph; keep `@tanisjam/peel` note |
| `.changeset/release-0-1-0.md` | Deleted (by `changeset version`) | Consumed; transformed into CHANGELOG entry |

## Interfaces / Contracts

### Changeset content

```markdown
---
"@tanisjam/peel": minor
---

First real MVP release.

- `peel init` — generate a minimal `.peel.yml` with detection of package manager, env files, scripts
- `peel run [branch] [mode]` — spin up a branch in an isolated git worktree with its own `node_modules`, env, and free port; auto-cleanup on exit
- `peel list` — table of peel-managed worktrees with branch, path, age, status
- `peel clean [branch] [--all] [--stale]` — remove tool-created worktrees safely (skips running on bulk modes)
- `peel config show|path|edit` — inspect, locate, or open the merged config
- Type-to-filter (autocomplete) branch picker
- SIGINT cleanup proven end-to-end via integration test

Available on npm as `@tanisjam/peel` (the bare name `peel` is squatted). The terminal binary is `peel`.
```

### README intro replacement

Drop the "Work in progress / placeholder" paragraph. Keep the one-line quote, the scoped-name explanation, and the install snippet. New intro:

```markdown
# peel

> Ephemeral git worktrees with auto-managed dev servers.

`peel` lets you spin up any branch in an isolated git worktree with its own
`node_modules`, a copied `.env`, and a free port — and tear it all down on Ctrl+C.

Published as `@tanisjam/peel` (the bare `peel` name is squatted). The binary on
your terminal is still `peel`.
```

### Post-merge release commands

```sh
# 1. Maintainer (manual):
git checkout main && git pull
npm publish --access public      # uses ~/.npmrc auth

# 2. Orchestrator (after maintainer reports success):
git tag -a v0.1.0 -m "v0.1.0 — first MVP release"
git push origin v0.1.0
gh release create v0.1.0 --title "v0.1.0" --notes-file <(awk '/^## 0\.1\.0/,/^## /' CHANGELOG.md | sed '$d')
```

## Testing Strategy

| Layer | What to Test | Approach |
|---|---|---|
| Gates | All existing tests still pass with bumped version | `npm run prepublishOnly` (lint + typecheck + test + build) |
| Smoke | The built binary reports the new version | `node dist/index.js --version` prints `0.1.0` after bump |
| Post-publish | The published package is installable | After `npm publish`, `npx -y @tanisjam/peel@0.1.0 --version` from a fresh shell prints `0.1.0` |

No new unit tests — the existing `src/version.test.ts` only asserts non-empty, which already passes.

## Migration / Rollout

No code migration. Users on `0.0.1` re-run `npm install -g @tanisjam/peel@latest` (or `npx @tanisjam/peel`).
