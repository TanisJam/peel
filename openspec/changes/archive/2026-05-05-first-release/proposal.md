# Proposal: first-release (0.1.0)

## Intent

Bump from `0.0.1` (placeholder published only to reserve the npm name) to `0.1.0` — the first real MVP release. The user-facing CLI surface is complete: `peel init`, `peel run`, `peel list`, `peel clean`, `peel config`, with proven SIGINT cleanup and a fuzzy branch picker. Time to publish.

## Scope

### In Scope
- Add a `minor` changeset describing 0.1.0 (init/run/list/clean/config + fuzzy + SIGINT-proven cleanup).
- Run `npx changeset version` → bumps `package.json` to `0.1.0` and generates `CHANGELOG.md`.
- Update `src/version.ts` to match (manual sync — see follow-up risk).
- Trim "Work in progress / placeholder" wording from the README intro; keep the `@tanisjam/peel` install snippet.
- Open PR for review.
- After merge + maintainer's `npm publish`: tag `v0.1.0`, push tags, create GitHub release with the CHANGELOG section.

### Out of Scope
- GitHub Actions release automation (`changesets/action`) — separate follow-up change.
- `npm publish` itself — maintainer runs locally with their npm auth.
- Auto-syncing `src/version.ts` from `package.json` at build time — defer.
- Coverage tooling (`@vitest/coverage-v8`) — recurring SUGGESTION, not blocking.

## Capabilities

### New Capabilities
None.

### Modified Capabilities
None.

> Pure metadata + packaging change. No behavioral impact.

## Approach

Manual changeset flow per exploration's Approach 1. Verify everything via `npm run prepublishOnly` (already wired with lint + typecheck + test + build) before opening the PR. After merge, the orchestrator runs `git tag v0.1.0 && git push --tags && gh release create v0.1.0` so the release reflects the actually-published commit.

## Affected Areas

| Area | Impact | Description |
|------|--------|-------------|
| `.changeset/<random>.md` | New | Source for the version bump + CHANGELOG entry |
| `package.json` | Modified | `version: "0.0.1" → "0.1.0"`; description drops "Work in progress" |
| `src/version.ts` | Modified | `VERSION = "0.1.0"` (manual sync) |
| `CHANGELOG.md` | New | Generated by `changeset version` |
| `README.md` | Modified | Trim placeholder wording in the intro |

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| `version.ts` drift from `package.json` | Med | Manual sync flagged in apply-progress; future change auto-syncs at build time |
| Unpublishable due to broken gates | Low | `prepublishOnly` runs lint+typecheck+test+build — same as CI matrix |
| README still references "placeholder" | Low | Explicit task in the apply phase to scrub |
| GitHub release missing | Low | Tag + release after maintainer confirms `npm publish` succeeded |

## Rollback Plan

Two layers. (a) Pre-publish: revert the merge commit; the placeholder `0.0.1` stays on npm untouched. (b) Post-publish: `npm deprecate '@tanisjam/peel@0.1.0' "<reason>"`; published versions on npm cannot be unpublished after 72 hours, so deprecation is the safe response. Future bumps to `0.1.1+` supersede.

## Dependencies

- npm auth on the maintainer's machine (`npm whoami` reports `tanisjam`).
- `gh` CLI authenticated for the release creation step.

## Success Criteria

- [ ] PR merges with all CI gates green.
- [ ] Maintainer runs `npm publish --access public` and reports success.
- [ ] `npx @tanisjam/peel@0.1.0 --version` prints `0.1.0` from a fresh shell.
- [ ] `git tag v0.1.0` exists on `origin/main` and a GitHub release at https://github.com/TanisJam/peel/releases/tag/v0.1.0 contains the CHANGELOG section.
- [ ] No follow-up "I forgot to update X" commits — README, CHANGELOG, version.ts, package.json all aligned.
